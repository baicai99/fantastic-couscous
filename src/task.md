# PRD（Ant Design 版）+ 任务拆分（按里程碑逐步完成）

> 说明：UI 组件库统一使用 **Ant Design（antd）**。以下将 PRD 拆成可落地的任务清单，按依赖关系组织；每个任务给出“完成定义（DoD）/验收点”。

---

## 0. 总体里程碑（建议顺序）
- M1：三栏骨架 + 对话数据模型 + 本地持久化（不含生成）
- M2：单窗口生成记录展示（网格）+ 灯箱预览交互
- M3：右侧设置区（单窗口）+ YAML 驱动能力渲染
- M4：A/B 对照模式（双窗口 + 单输入 + 严格对齐）
- M5：A/B 联动预览 + 失败摘要 + 重试
- M6：动态注入（{{key}}）三种输入方式
- M7：打磨与增强项（列表批量/保存规则/对照扩展等）

---

## 1. 基础工程与 UI 框架（M1-Task 1）
### T1.1 工程初始化（React + antd）
- 内容
  - 初始化 React 项目
  - 集成 Ant Design（基础主题、全局样式）
  - 规划路由（可选，本期可单页）
- DoD / 验收
  - 页面能正常渲染 antd Button / Layout / Typography
  - 无运行时报错

### T1.2 三栏布局骨架（antd Layout）
- 内容
  - 使用 `Layout` / `Sider` / `Content` 搭建三栏
  - 左：对话列表区（含顶部“新建对话”按钮）
  - 中：对话区（含空状态 LOGO 占位）
  - 右：设置区占位（先放 Card + Form 占位）
- DoD / 验收
  - 三栏固定存在，左右可滚动，中间为主内容区
  - 无对话时中间显示 LOGO 占位（可先用 placeholder 图）

## 2. 对话数据模型与本地持久化（M1-Task 2）
### T2.1 数据结构定义（对话/消息/生成批次/图片项）
- 内容
  - 定义最小数据模型：
    - Conversation
    - Message（含：user prompt、assistant run 记录）
    - Run（批次）与 ImageItem（序号、文件引用、状态）
    - Side（single/A/B；在 A/B 下标记 A 或 B）
- DoD / 验收
  - 能在内存创建多对话、切换对话、追加消息
  - 数据结构可序列化

### T2.2 本地存储：对话元数据与索引
- 内容
  - 使用浏览器本地存储保存：
    - 对话列表索引
    - 对话内容（messages / runs）
  - 提供“加载/保存”封装
- DoD / 验收
  - 刷新页面后对话列表与内容仍存在
  - 新建/切换/追加消息后会持久化

> 注：图片文件“真正落盘”若使用纯 Web，会受限于浏览器沙盒。MVP 可先保存为 base64/blob 到 IndexedDB；后续如有桌面壳（Tauri/Electron）再做真实文件系统落盘。

## 3. 中间对话区：消息流 + 网格展示（M2-Task 3）
### T3.1 聊天流渲染（单窗口）
- 内容
  - 中间区渲染消息列表
  - 用户输入消息展示为一条气泡
  - assistant 生成结果展示为一条气泡（先用 mock 数据）
- antd 建议组件
  - `Card`/`List`/`Space`/`Typography`/`Divider`
- DoD / 验收
  - 有消息时不显示空状态 LOGO
  - 消息顺序稳定（时间线）

### T3.2 生成结果网格（气泡内 Grid）
- 内容
  - 单次 run 生成 N 张时，在同一气泡内以网格展示
  - 网格规则：优先接近方形（N=4 为 2×2）
- antd 建议组件
  - `Row`/`Col` 或 `Image.PreviewGroup`（仅用于预览组）
  - `Skeleton.Image`（加载态）
- DoD / 验收
  - N=4 显示 2×2
  - 图片顺序稳定（按序号 1..N）

## 4. 灯箱预览交互（M2-Task 4）
### T4.1 单侧灯箱（基础版）
- 内容
  - 点击图片进入预览
  - 支持：左右按钮、键盘 ←/→、ESC 退出
- antd 建议组件
  - `Image` 的 `preview` 或自建 `Modal` + `Carousel`
- DoD / 验收
  - 上述交互都可用，切换与退出无异常

### T4.2 缩放/平移交互（Ctrl+滚轮、Ctrl+拖拽）
- 内容
  - 预览内支持：
    - Ctrl + 滚轮缩放
    - Ctrl + 鼠标左键拖拽平移
- DoD / 验收
  - 缩放/平移只在预览内生效，不影响页面滚动
  - 退出预览后状态复位（不影响下一次预览）

## 5. 右侧设置区（单窗口）+ API 渠道管理（M3-Task 5）
### T5.1 设置区基础表单（单窗口）
- 内容
  - 分辨率、长宽比、一次生成张数、自动保存本地（开关）
- antd 建议组件
  - `Form`/`Select`/`InputNumber`/`Switch`/`Card`
- DoD / 验收
  - 用户调整设置后能写入当前对话/当前侧配置（single）

### T5.2 API 渠道管理（最小可用）
- 内容
  - 可维护多个渠道（名称、BaseURL、Key 等“字段存在即可”）
  - 选择当前渠道用于生成
- antd 建议组件
  - `Drawer`/`Modal` + `Form.List` + `Table`
- DoD / 验收
  - 可新增/编辑/删除渠道（本地保存）
  - 可选择当前渠道（生效可追溯：记录到 run）

## 6. YAML 驱动：模型能力渲染（M3-Task 6）
### T6.1 YAML 结构加载与解析
- 内容
  - 本地内置一份 YAML（后续支持外部编辑）
  - YAML 描述：
    - 模型列表
    - 每模型支持哪些参数、默认值、范围/枚举
- DoD / 验收
  - 能从 YAML 读出模型与能力结构并渲染到 UI

### T6.2 参数表单“按模型裁剪”
- 内容
  - 切换模型后，右侧只显示该模型支持的参数项
  - 不支持项隐藏或置灰并提示“不支持”
- DoD / 验收
  - 用户不会设置到无效参数
  - run 记录含：模型名 + 生效参数快照

## 7. A/B 对照模式（双窗口 + 单输入）（M4-Task 7）
### T7.1 对照模式开关与双窗口布局
- 内容
  - 开启对照组：中间区从单窗口变左右双窗口（A/B）
  - 输入框保持一个
- antd 建议组件
  - `Tabs`（不建议放中间，容易破坏“同时可见”）
  - `Grid`/`Row`/`Col` 将两个对话窗并排
- DoD / 验收
  - 两个对话窗口同时可见且各自滚动不互相影响
  - 关闭对照组回到单窗口

### T7.2 右侧设置区 A/B Tab
- 内容
  - 右侧设置区顶部提供 A/B Tab
  - 切换 Tab 编辑对应侧的：渠道/模型/参数/（可选）提示词覆盖
- antd 建议组件
  - `Tabs` + `Form`
- DoD / 验收
  - A 与 B 的设置互不污染
  - 发送时两侧使用各自配置

### T7.3 严格对齐展示（同批次同序号）
- 内容
  - 同一次发送，A/B 生成结果形成“同批次编号”
  - 展示时按批次对齐；同序号图片对齐
  - 缺失用占位并展示原因
- DoD / 验收
  - 任何一侧失败/缺图都不会导致错位对比

---

## 8. A/B 联动预览 + 失败摘要 + 重试（M5-Task 8）
### T8.1 联动预览（同序号双侧同时显示）
- 内容
  - A/B 模式下点击预览：同时展示 A 与 B 的同序号图片
  - 左右切换时联动切换同序号
  - 缩放/平移尽量联动一致
- DoD / 验收
  - 能一眼对比同序号的 A/B 输出
  - 切换不会串图/错位

### T8.2 失败摘要（B 级信息）
- 内容
  - 失败时显示原因摘要（超时/鉴权/限流/参数不支持/拒绝等）
- DoD / 验收
  - 用户可理解失败原因并定位到具体 run/具体侧

### T8.3 失败重试（保留原配置）
- 内容
  - 对失败项提供重试按钮
  - 重试复用当时的提示词与参数快照
- DoD / 验收
  - 重试可追溯（记录重试次数与结果）
  - 不会用到“当前最新设置”误覆盖历史

---

## 9. 动态注入（{{key}}）三种方式（M6-Task 9）
### T9.1 模板识别与替换引擎
- 内容
  - 解析 `{{key}}`
  - 将变量输入替换为最终提示词
  - 保存：模板/变量/最终提示词
- DoD / 验收
  - 缺 key、多余 key、有空值时有明确提示（阻止或警告要一致）

### T9.2 三种输入方式 UI（D：三者都要）
- 内容
  1) 表格方式（key 固定填 value，可扩展多组）
  2) 内联 `k=v` 解析
  3) 变量面板（定义 + 值集合）
- antd 建议组件
  - `Table` + `Form.List`、`Input.TextArea`、`Collapse`/`Tabs`
- DoD / 验收
  - 至少有一个默认入口最顺手（建议：表格为默认）
  - 三种方式最终都走同一替换引擎，行为一致
